# 여행플랜 · 구글 플레이스토어 앱 등록 플랜

> 작성일: 2026-07-18
> 대상: 현재 `travel_plan` 웹앱(Vite + React SPA)을 Google Play 스토어에 앱으로 등록하기

---

## 0. 한 줄 결론

**현재 코드(웹앱) 파일 그 자체를 플레이스토어에 올릴 수는 없습니다.**
플레이스토어는 웹 URL이 아니라 **안드로이드 앱 패키지(`.aab`/`.apk`)**만 받습니다.
하지만 이 웹앱을 **얇은 안드로이드 껍데기로 감싸면(래핑)** 등록 가능하고,
이 프로젝트는 이미 PWA 기반(모바일 UI, `manifest.json`, `standalone` 모드)이 갖춰져 있어 **비교적 쉽게 가능**합니다.

- 추천 방식: **TWA (Trusted Web Activity)** — 배포된 웹사이트를 그대로 전체화면 앱으로 감싸는 방식
- 예상 순수 작업량: **1~3일** (코드 보완 + 패키징 + 스토어 등록 서류)
- 비용: **Google Play 개발자 등록비 1회 $25**

## 1. 현재 앱 구조 분석

| 항목 | 현재 상태 | 플레이스토어 관점 |
|------|-----------|-------------------|
| 프레임워크 | Vite 8 + React 19 (SPA) | 순수 웹 → 그대로는 등록 불가, 래핑 필요 |
| 라우팅 | `react-router-dom` (클라이언트 라우팅) | TWA/WebView와 호환 OK |
| 백엔드 | Supabase (Postgres + Edge Functions 인증) | 서버가 클라우드에 있어 앱에서 그대로 호출 가능 OK |
| 배포 | Vercel (`vercel.json` SPA rewrite) | 공개 HTTPS URL 필요 → **아래 2-A 주의** |
| PWA manifest | `public/manifest.json` 존재 (`display: standalone`, `portrait`) | TWA에 필수 요소 → 있음 ✅ (단, 아이콘 보완 필요) |
| 모바일 메타 | viewport 고정, `theme-color`, `mobile-web-app-capable` 등 | 모바일 앱처럼 이미 세팅됨 ✅ |
| 서비스워커 | **없음** (오프라인/설치가능성 미지원) | PWA 설치 기준 미달 → **추가 권장** |
| UI | 하단 탭 네비게이션, 세로 고정, 모바일 우선 | 이미 앱처럼 보임 ✅ |
| 인증 | 자체 `users` 테이블 + SHA-256 (Edge Function 검증) | 개인정보 수집 → **개인정보처리방침 필수** |
| 수집 데이터 | 아이디/비밀번호/닉네임/여행사진/비용 | Play "데이터 보안" 양식 작성 필요 |

**요약:** 모바일 앱 형태의 UI와 PWA 기본기는 갖췄으나, ①안드로이드 패키지가 없고 ②서비스워커/아이콘 등 PWA 완성도가 부족하며 ③스토어 정책용 서류(개인정보처리방침 등)가 없습니다.

## 2. 등록을 막는 요소 (반드시 먼저 해결)

### 2-A. 🚨 사이트가 공개 접근 가능해야 함 (가장 중요)
Git 이력에 `fix: add crossorigin to manifest to fix vercel sso cors` 커밋이 있습니다.
이는 배포가 **Vercel SSO/비밀번호 보호(로그인 벽) 뒤에 있을 가능성**을 시사합니다.

- TWA/WebView 앱은 배포 URL을 그대로 로드하므로, **로그인 벽이 있으면 앱이 흰 화면/차단**됩니다.
- Google의 도메인 소유 검증(Digital Asset Links)도 **공개 접근 가능한 URL**이어야 통과합니다.
- ✅ 조치: Vercel 프로젝트의 **Deployment Protection(SSO/Password)을 해제**하거나, 별도의 공개 도메인으로 배포.

### 2-B. 안정적인 커스텀 도메인 권장
TWA는 특정 도메인에 **서명키를 묶어(Asset Links)** URL 주소창을 숨깁니다.
Vercel 기본 도메인(`*.vercel.app`)도 되지만, 배포마다 URL이 바뀌면 곤란하므로 **고정 도메인(예: `travelplan.앱도메인.com`)** 사용을 권장합니다.

### 2-C. 개인정보처리방침(Privacy Policy) URL
아이디/비밀번호/사진 등 개인정보를 수집하므로 Play 정책상 **필수**입니다.
간단한 방침 페이지를 웹앱 내(`/privacy`) 또는 별도 URL로 준비해야 합니다.

## 3. 등록 방식 3가지 비교

| 방식 | 설명 | 난이도 | 네이티브 기능 | 추천도 |
|------|------|--------|----------------|--------|
| **① TWA (Trusted Web Activity)** | 배포된 웹사이트를 Chrome 엔진으로 전체화면 렌더링하는 얇은 껍데기. `PWABuilder` 또는 `Bubblewrap`으로 생성 | ★☆☆ 낮음 | 제한적(웹 API 범위) | ⭐ **최우선 추천** |
| **② Capacitor (WebView 래핑)** | 웹 빌드 결과물을 앱 안에 넣고 WebView로 구동. 카메라/푸시 등 네이티브 플러그인 사용 가능 | ★★☆ 중간 | 풍부(플러그인) | 네이티브 기능 필요 시 |
| **③ React Native / 네이티브 재작성** | 앱을 처음부터 다시 개발 | ★★★ 높음 | 완전 | 지금은 불필요 |

### 왜 TWA를 추천하나
- 이미 **배포된 웹사이트를 그대로** 사용 → 코드 재작성 거의 없음
- 웹앱을 업데이트하면 **앱도 자동으로 최신** (스토어 재심사 없이 웹만 배포하면 됨)
- 이 프로젝트처럼 "웹이 메인, 앱은 배포 채널"인 경우 최적

### TWA의 한계 (참고)
- 카메라/파일 접근 등은 **웹 표준 API 범위 내에서만** 동작 (사진 업로드는 `<input type=file>`로 가능하므로 이 앱은 문제 없음)
- 푸시 알림 등 고급 네이티브 기능이 필요해지면 그때 ②Capacitor로 전환 고려

## 4. 권장 경로: TWA 상세 체크리스트

### STEP 1. 웹앱(PWA) 완성도 보완 — *코드 작업*
- [ ] **서비스워커 추가**: `vite-plugin-pwa` 도입 (설치가능성 + 오프라인 캐시 + PWABuilder 점수 통과)
- [ ] **아이콘 정비**: 현재 `manifest.json`은 `favicon.png` 하나를 `192x192 512x512`로 선언 → **실제 192px, 512px PNG 분리 제작** + **maskable 전용 아이콘**(안전영역 여백 포함) 별도 준비
- [ ] `manifest.json` 점검: `name`, `short_name`, `start_url`, `display: standalone`, `theme_color` (대부분 이미 OK)
- [ ] 스플래시/상태바 색상 확인 (`theme_color`, `background_color`)
- [ ] 배포 후 **Lighthouse PWA 검사** 통과 확인

### STEP 2. 공개 배포 + 도메인 — *인프라 작업*
- [ ] Vercel **Deployment Protection 해제** (2-A) → 로그인 없이 접속되는지 확인
- [ ] 고정 도메인 연결 (권장)
- [ ] HTTPS 정상 동작 확인

### STEP 3. 안드로이드 패키지 생성 — *패키징*
두 가지 툴 중 택1:
- **PWABuilder** (웹 UI, 가장 쉬움): https://www.pwabuilder.com 에 배포 URL 입력 → Android 패키지(`.aab`) 다운로드
- **Bubblewrap** (CLI, 세밀한 제어):
  ```bash
  npm i -g @bubblewrap/cli
  bubblewrap init --manifest https://<도메인>/manifest.json
  bubblewrap build   # app-release-signed.aab 생성
  ```
- [ ] 앱 서명키(keystore) 생성 및 **안전하게 백업** (분실 시 앱 업데이트 불가)
- [ ] 패키지의 applicationId(예: `com.travelplan.app`) 결정

### STEP 4. 도메인 소유 검증 (URL 주소창 숨기기) — *연동*
- [ ] 서명키의 SHA-256 지문으로 `assetlinks.json` 생성
- [ ] 웹 서버의 `/.well-known/assetlinks.json` 경로에 배치 (Vercel `public/.well-known/assetlinks.json`)
- [ ] 배포 후 `https://<도메인>/.well-known/assetlinks.json` 접근 확인 → 이게 있어야 앱에 브라우저 주소창이 안 뜸

### STEP 5. Play Console 등록 — *스토어 서류*
- [ ] **Google Play 개발자 계정 생성** (1회 $25 결제, 본인확인 절차 있음, 수일 소요 가능)
- [ ] 앱 만들기 → `.aab` 업로드
- [ ] 스토어 등록정보: 앱 이름/설명/스크린샷(폰 최소 2장)/피처 그래픽(1024×500)/아이콘(512×512)
- [ ] **개인정보처리방침 URL** 등록 (2-C)
- [ ] **데이터 보안(Data Safety)** 양식: 수집 항목(아이디/닉네임/사진/비용) 신고
- [ ] **콘텐츠 등급** 설문
- [ ] 대상 국가/연령, 광고 포함 여부(없음)
- [ ] 내부 테스트 → 비공개 테스트 → 프로덕션 순으로 출시 (신규 개인 계정은 **비공개 테스트 12명·14일** 요건이 붙을 수 있음 — 최신 정책 확인 필요)

## 5. 현재 코드에서 "바로 손봐야 할 것" 우선순위

| 우선순위 | 작업 | 이유 |
|:---:|------|------|
| 🔴 높음 | Vercel 접근 보호 해제 / 공개 URL 확보 | 이게 안 되면 앱이 아예 안 뜸 |
| 🔴 높음 | 개인정보처리방침 페이지 작성 | Play 정책상 미제출 시 반려 |
| 🟠 중간 | `vite-plugin-pwa`로 서비스워커 추가 | PWA/TWA 품질 기준 통과 |
| 🟠 중간 | 아이콘 192/512/maskable 분리 제작 | 아이콘 깨짐·심사 감점 방지 |
| 🟢 낮음 | 비밀번호 해시 보강(솔트/서버 bcrypt) | 스토어 필수는 아니나 보안상 권장 |

> 참고: 현재 인증은 SHA-256 단순 해시입니다. 스토어 등록의 **차단 요소는 아니지만**, 개인정보를 다루는 앱이므로 장기적으로 서버측 솔트+bcrypt 등으로 강화하는 것을 권장합니다.

## 7. 다음 액션 제안

가장 먼저 확인이 필요한 것은 **"배포 URL이 로그인 없이 공개로 열리는가"** 입니다.
이것만 정리되면 나머지는 순차적으로 진행 가능합니다.

원하시면 다음 중 무엇이든 이어서 진행해 드릴 수 있습니다:
1. `vite-plugin-pwa` 도입 + 서비스워커/아이콘 세팅 (코드 작업)
2. 개인정보처리방침 페이지(`/privacy`) 초안 작성
3. `public/.well-known/assetlinks.json` 자리 잡기 + PWABuilder용 manifest 점검
4. Vercel 공개 배포/도메인 설정 안내
