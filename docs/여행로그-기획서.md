# 여행플랜 고도화 기획서 — 여행 로그(Travel Log)

## 1. 개요

### 목적
현재 앱은 "여행을 **계획**하는" 도구에 머물러 있음. 여행이 끝나면 방의 활용도가 사라짐.
**여행 로그** 기능으로 "여행 후"의 경험(추억 아카이빙)까지 담아, 여행 전 → 중 → 후의 전체 라이프사이클을 커버한다.

### 핵심 컨셉
- 여행이 완료된 방은 **여행 로그**에 아카이빙됨
- 여행 로그는 **세계지도 → 국가 상세지도**로 이어지는 "지리적 드릴다운(줌인)" 방식으로 탐색
  - ① 세계지도에 다녀온 **국가별 다트**가 표시됨
  - ② 국가 다트를 클릭하면 **그 나라로 줌인되는 애니메이션** 후 국가 상세지도로 전환
  - ③ 상세지도의 **지역 다트**를 클릭하면 그 여행의 **계획 요약 + 실제 다녀온 사진** 표시
- 예: 세계지도 → 대한민국 다트 클릭 → 지도가 확대되며 한국 지도로 전환 → 부산 다트 클릭 → 부산 여행 로그
- **노출 범위: 내가 멤버로 참여한 완료 여행만 보인다.** (로비의 "나의 여행 방"과 동일한 스코프 — 다른 그룹의 여행은 보이지 않음)

## 2. 기능 상세

### 2-1. 여행 로그 진입점 (로비)
- 로비의 기존 두 버튼(`새 여행 방 만들기`, `초대 코드로 입장`) 옆/아래에 **`📖 여행 로그`** 버튼 추가
- 클릭 시 여행 로그 페이지로 이동 (`/travel-log`)
- ⚠️ 라우팅 구조: `/travel-log`는 특정 방에 종속되지 않으므로 **`App.jsx`의 최상위 라우트**(`/lobby`, `/mypage`와 동일 레벨)로 추가한다. `TripLayout` 하위가 아니므로 **`useTripContext`를 쓸 수 없고**, 여행 로그와 로그 상세는 각자 필요한 데이터(트립·멤버 이름 등)를 자체적으로 조회해야 한다.

### 2-2. 여행 로그 페이지 (세계지도 기반 드릴다운)
탭 구분 없이, 하나의 지도 화면에서 줌인으로 탐색한다.

- **레벨 1 — 세계지도 (기본 화면, 내가 참여한 완료 여행만 표시)**
  - 세계지도 위에 여행 다녀온 **국가마다 다트(🎯)** 표시 (범위는 §5-1 참고 — 전체 국가가 아닌 우선 국가 목록)
  - 같은 나라를 여러 번 갔으면 다트 1개 + 뱃지로 횟수 표시(예: `×3`)
  - 국가 다트 클릭 시:
    - **상세지도 지원 국가(MVP: 대한민국)** → **줌인 애니메이션** 후 레벨 2로 진입
    - **상세지도 미지원 국가** → 그 나라에서 완료한 여행이 1개면 바로 로그 상세로, **2개 이상이면 방 이름 목록에서 선택**하는 팝업 후 로그 상세로 진입
- **레벨 2 — 국가 상세지도 (줌인 후)**
  - 세계지도가 해당 국가로 확대되며 상세지도로 전환 (예: 대한민국 → `KoreaMapSVG` 재활용)
  - 상세지도 위에 그 나라에서 다녀온 **도(道)/광역시 단위 다트** 표시
    - ⚠️ **중요**: 현재 `KoreaMapSVG`는 다트를 **도(道) 중심 좌표**에만 찍을 수 있음(시/군 좌표 없음). 따라서 "부산 다트", "강원 다트"처럼 **도 단위로 한 개씩** 찍히고, 같은 도에 여러 여행(예: 강원 속초 + 강원 강릉)이 있으면 **다트 1개 + 뱃지(예: `×2`)** 로 묶인다.
  - 도 다트 클릭 시: 그 도의 완료 여행이 1개면 바로 여행 로그 상세(§2-3)로, **2개 이상이면 방 이름 + 세부 지역명(예: "제주 여름휴가 · 서귀포") 목록에서 선택**하는 팝업 후 로그 상세로 진입
  - "← 세계지도로" 버튼(줌아웃)으로 레벨 1 복귀
- **줌 애니메이션**
  - 세계지도에서 선택 국가 위치로 부드럽게 확대되다가 국가 상세지도로 크로스페이드/전환
  - 뒤로 갈 때는 역방향(줌아웃)
- **빈 상태(empty state)**: 완료된 여행이 0건이면 세계지도(또는 안내 일러스트)와 함께 "아직 완료된 여행이 없어요. 여행을 마치면 여기에 기록돼요" 안내를 표시한다.

> ⚠️ 큰 전제: 현재는 **한국 상세지도(`KoreaMapSVG`)만 존재**한다. 세계지도 자산, 줌 애니메이션, 그리고 한국 외 국가의 상세지도는 모두 신규 제작이 필요하다(§5 참고). 이 때문에 **한국 외 국가는 상세지도 없이 다트 클릭 시 바로 로그(또는 그 나라 여행 리스트)로 진입**하는 방식으로 단계적 도입을 권장한다.

### 2-3. 여행 로그 상세 (지역 다트 클릭 시)
클릭하면 해당 방의 상세 로그가 열림. 두 블록으로 구성:
- **① 계획 요약 (자동 생성, 읽기 전용)** — 기존 데이터를 모아서 보여줌
  - 여행 기간 (`trips.start_date` 기반)
  - 일정 요약 (`plans` — Day별 주요 일정)
  - 총 지출 / 1인당 (`expenses`)
  - 확정 숙소 (`accommodations` 중 `is_selected`)
  - 준비물 완료율 (`checklist`)
- **② 여행 사진 (신규)**
  - 실제 다녀온 후 찍은 사진 업로드 (다중 업로드)
  - 방 멤버 누구나 추가 가능, **방(여행)당 최대 10장**, 갤러리(그리드) 형태로 표시
  - **업로드 전 클라이언트 단에서 리사이즈/압축 처리** (원본 폰 사진 그대로 올리지 않도록 — 가로/세로 최대 크기 제한 + JPEG 압축 후 업로드)
  - **업로더 식별**: `trip_photos.uploaded_by`에는 (기존 `expenses`/`plans` 패턴과 동일하게) `user.id`를 저장하고, 표시 이름은 조회한 멤버 정보로 변환한다.
  - **사진 삭제 권한**: 사진을 올린 본인 또는 방장만 삭제 가능(`canEditItem` 유틸 패턴 재사용).
  - **삭제 시 Storage 정리**: DB 행뿐 아니라 **Storage 실제 파일도 함께 삭제**한다. (현재 `AccommodationPage.handleDelete`는 DB 행만 지우고 Storage 파일은 남겨 고아 파일이 쌓이는 상태 — 용량이 이 기능의 핵심 동기이므로 여행 사진은 이 문제를 처음부터 방지한다.)

### 2-4. 여행 완료 버튼
- **위치 추천: 홈(Dashboard) 상단 헤더** — 방 이름/날짜 수정 버튼이 이미 있는 그 영역에 **방장에게만** 보이는 `✅ 여행 완료` 버튼
  - 이유: 홈이 방의 대표 화면이고 방 관리성 액션(이름·날짜 변경)이 모여있어 맥락이 자연스러움. 로비 방 카드에 두는 대안도 있으나, 완료 시 지역·후기 입력이 필요해 방 내부가 더 적합.
- **완료 플로우 (버튼 클릭 시 모달)**:
  1. **국가 선택** (기본값: 대한민국)
  2. 대한민국이면 세부 지역 선택 — `KOREA_REGIONS` 재활용 / 그 외 국가면 도시·지역 텍스트 입력
  3. (선택) 한 줄 여행 후기
  4. 완료 확정
  - → 저장된 국가 정보로 세계지도 다트가, 지역 정보로 (상세지도 지원 국가의) 지역 다트가 찍힘
- **완료 후 상태 변화**:
  - 홈 상단과 로비 방 카드에 `✅ 여행 완료` 뱃지 표시
  - 여행 로그에 자동 등록되어 지도/리스트에 나타남 (완료된 방만 노출, 내가 멤버로 참여한 여행만)
  - **완료 취소**: 방장은 `✅ 여행 완료` 뱃지 자리에서 "완료 취소" 가능. 취소하면 여행 로그에서 제외되고 홈은 다시 진행 중 상태(완료 버튼 노출)로 돌아감. 이때 **이미 올린 여행 사진·후기는 삭제하지 않고 유지**한다(재완료 시 그대로 다시 보임).
- **방장이 방을 나간 경우**: 방장이 "나가기"를 하면, **남은 멤버 전원이 방장 권한(완료/완료취소 포함)을 갖게 된다.** 방장 권한자가 아무도 없는 상태가 되지 않도록 하는 안전장치.

             │     └─ [레벨 2] 국가 상세지도 + 지역별 다트   (← 세계지도로: 줌아웃)

             └─ 상세지도 미지원국 다트 클릭 → (동일 국가 2개 이상이면 방 이름 선택) → 여행 로그 상세

방 안 > 홈(Dashboard)
 └─ ✅ 여행 완료 버튼(방장) → 완료 모달(국가·지역·후기) → 완료 처리
    (방장이 나가면 남은 멤버 전원이 방장 권한 획득)

## 4. 데이터 모델 변경 (필요 항목)

| 대상 | 변경 | 설명 |
|---|---|---|
| `trips` | `is_completed` (bool), `completed_at` (timestamp) | 여행 완료 여부/시점 |
| `trips` | `country_code` (신규 `SUPPORTED_COUNTRIES` 목록 기준 코드) | 세계지도 국가 다트 위치 |
| `trips` | `region_id` (대한민국이면 `KOREA_REGIONS`의 id), `region_label` (지역명 텍스트, `region_id`와 함께 저장), `region_province` (상세지도 배치용 도) | 국가 상세지도 지역 다트 위치. `region_label`은 `KOREA_REGIONS` 목록이 나중에 바뀌어도 표시가 깨지지 않도록 값 자체를 함께 저장 |
| `trips` | `destination_label` (도시·지역 자유 텍스트) | 상세지도 미지원 국가 표기용 |
| `trips` | `review_note` (한 줄 후기, 선택) | 완료 시 입력 |
| `trip_members` | `is_admin` (bool, 기본 false, 방 생성자만 true) | **다중 방장 지원.** 방장이 나가면(§2-4) 남은 멤버 전원의 `is_admin`을 true로 전환 |
| 신규 `trip_photos` | `id`, `trip_id`, `url`, `uploaded_by`, `created_at` | 여행 후 사진 (다중, 업로드 전 클라이언트에서 리사이즈/압축) |
| 신규 `src/data/countries.js` | `SUPPORTED_COUNTRIES` (국가 코드, 한글명, 세계지도 다트 좌표) | `KOREA_REGIONS`와 동일한 패턴의 정적 데이터. 여행이 불가능한 국가(북한 등) 제외, 방문 빈도가 높은 국가 위주로 우선 구성 (§5-1) |
| Storage | 기존 `travel_images` 버킷 재활용 | 숙소 사진 업로드 로직 재사용 가능 |

> 참고: 현재 홈의 다트 게임은 "랜덤 추천"일 뿐 방에 지역을 저장하지 않음. 그래서 지도에 완료 여행을 찍으려면 **완료 시점에 국가·지역을 확정·저장**하는 절차가 반드시 필요함.
> 참고: `admin_id`(방 생성자, 단일값)는 기존 표시용으로 유지하고, 실제 "방장 권한이 있는가" 판정은 `trip_members.is_admin`으로 옮겨간다 (다중 방장 지원을 위해).
> ⚠️ **기존 데이터 마이그레이션 필수**: `trip_members.is_admin` 컬럼을 추가할 때, 이미 존재하는 방들에 대해 **`trips.admin_id`와 일치하는 멤버의 `is_admin`을 true로 백필(backfill)** 해야 한다. 그렇지 않으면 기존 방들은 방장이 아무도 없는 상태가 되어 완료 처리를 못 한다.
> ⚠️ **`KOREA_REGIONS` 도(道) 커버리지 공백**: 현재 `KOREA_REGIONS`에는 **대전·세종·대구·광주** 지역이 없음(§5-12). 이 도들로 여행한 경우 완료 모달에서 고를 항목이 없으므로, (a) `KOREA_REGIONS`에 해당 도의 지역을 추가하거나 (b) 완료 모달에서 도 단위 직접 선택을 허용하는 보완이 필요하다.

## 5. 기술 고려사항 & 결정 사항

1. **세계지도 자산 (신규, 최대 난관)** — 현재 세계지도 컴포넌트가 없음. SVG 세계지도(국가별 path)와 각 국가의 대략적 좌표(다트 위치)가 필요. 한국 지도처럼 정적 SVG path를 사전 생성하는 방식(`generate_paths.js` 패턴) 재활용.
   - **결정: 전체 195개국이 아니라 우선순위 국가 목록으로 시작.** 여행이 불가능한 국가(예: 북한 등 여행 금지/제한국)는 제외하고, 실제로 많이 방문하는 국가(동아시아·동남아시아·미주·유럽 주요국 등) 위주로 `SUPPORTED_COUNTRIES` 목록을 구성한다. 목록에 없는 국가로 완료 처리를 하는 경우의 예외 처리(예: "기타" 처리 또는 목록 확장 요청)는 구현 단계에서 정한다.
   - 디테일 수준은 **홈 화면 한국 지도(`KoreaMapSVG`) 정도**로 맞춘다 (국가 경계가 뚜렷이 구분되는 수준. 아주 정밀한 지형 디테일까지는 필요 없음).
2. **줌인 애니메이션** — **결정: 크로스페이드 방식.** 세계지도에서 선택 국가 다트 위치로 확대되는 느낌을 준 뒤, 국가 상세지도로 크로스페이드 전환한다. 뒤로 갈 때(줌아웃)는 역방향으로 재생.
3. **국가 상세지도** — **결정: MVP는 대한민국만 상세지도로 드릴다운 지원.** 그 외 국가는 국가 다트 클릭 시 바로 로그(또는 그 나라 여행 리스트)로 진입한다. 다른 국가 상세지도는 추후 순차 제작.
4. **국내 지역 다트 다중 표시** — 현재 `KoreaMapSVG`는 다트 1개(단일 `result`)를 **도(道) 중심 좌표**에 찍는 구조라, 여러 다트를 동시에 받도록 컴포넌트 구조 변경이 필요함.
   - **결정: 다트는 도(道) 단위로 표시**(시/군 좌표가 없으므로). 같은 도에 여행이 여러 개면 다트 1개 + 뱃지로 묶고, 클릭 시 방 선택 팝업(§5-11).
   - **결정: 여러 다트 동시 표시의 실제 구현 가능 여부는 4차 착수 시 `KoreaMapSVG` 구조를 먼저 검토해 기술 검증(스파이크)한다.** 서로 다른 도의 다트는 겹치지 않으므로, 핵심 검증 포인트는 "단일 result 구조 → 다중 마커 구조로의 변경 난이도"임.
5. **지역 단위** — **지도 다트 배치는 도 단위(`province`)** 로만 가능(시/군 좌표 없음). 단, 로그 상세·방 선택 팝업의 텍스트에는 시/군(`region_label`)을 함께 보여준다. 그래서 `region_id`/`region_label`(시/군)과 `region_province`(도)를 모두 저장한다(§4).
6. **여행 완료 권한/되돌리기** — **결정: 완료 처리는 방장만 가능. 완료 취소도 방장만 가능하도록 허용한다.** (완료 취소 시 여행 로그에서도 제외되고 홈에 다시 진행 중 상태로 돌아감)
7. **사진 업로드 권한 & 용량** — **결정: 방(여행)당 최대 10장.** 업로드 권한은 기존 제안대로 멤버 누구나 가능(단, 합산 10장 제한에 도달하면 더 이상 추가 불가). **추가 결정: 업로드 전 클라이언트에서 리사이즈/압축**을 거쳐 원본 그대로 저장되지 않도록 한다 (Storage 용량 절약).
8. **완료 안 해도 로그를 보고 싶은가** — **결정: 완료된 방만 여행 로그에 노출.** 진행 중인 방은 로그에 나타나지 않는다.
9. **여행 로그 노출 범위** — **결정: 내가 멤버로 참여한 완료 여행만 노출.** 로비의 "나의 여행 방"과 동일한 스코프로, 다른 그룹의 여행은 보이지 않는다.
10. **방장 이탈 시 처리** — **결정: 방장이 방을 나가면 남은 멤버 전원이 방장 권한(완료/완료취소 포함)을 갖는다.** 데이터 모델은 `trip_members.is_admin`(다중 방장) 도입으로 지원한다(§4).
11. **동일 도(道)/국가에 여행이 여러 번인 경우** — **결정: 다트 클릭 시 해당 도(道)/국가의 완료 여행이 2개 이상이면 방 이름(+세부 지역명) 목록에서 하나를 선택**하게 하고, 선택 후 그 여행의 로그 상세로 진입한다. 1개면 바로 진입. (지도 다트가 도 단위이므로, 같은 도의 서로 다른 시/군 여행들도 이 팝업에 함께 묶여 나온다.)
12. **`KOREA_REGIONS` 도 커버리지 공백** — 현재 목록에 **대전·세종·대구·광주**가 없어 해당 도 여행은 완료 모달에서 지역을 고를 수 없음. → **결정 필요**: (a) 목록에 이 도들의 지역 추가, 또는 (b) 완료 모달에서 도 단위 직접 선택 허용. (권장: 최소한 도 단위 선택 fallback을 두어 어떤 국내 여행도 로그에 남길 수 있게 함)

> **인지된 한계 (의도적으로 남겨두는 리스크)**: `trips`/`trip_members` 테이블은 RLS가 꺼져있고 "방장만 가능" 같은 권한 체크가 클라이언트(JS)에서만 이뤄진다. anon key를 가진 사람이 직접 API를 호출하면 완료 처리·사진 업로드 제한(10장)·삭제 권한 등을 우회할 수 있다. 또한 10장 제한은 클라이언트가 현재 개수를 세어 막는 방식이라 여러 명이 동시에 올리면 잠깐 초과될 수 있다. 이는 이 앱 전체에 이미 적용되어 있는 트레이드오프(소규모 내부 앱 전제)와 동일한 성격이라 이번 기능에서 별도로 막지는 않는다.

## 6. 단계별 구현 제안 (승인 시 이 순서로)

- **1차** — 데이터 모델 추가(`trips` 컬럼 + `trip_photos` + `trip_members.is_admin`, **기존 데이터 `is_admin` 백필 포함**) & 여행 완료/완료취소 버튼·모달(홈, 국가·지역 선택, 방장 전용, 도 커버리지 공백 보완 §5-12) & 방장 이탈 시 전원 승격 로직
- **2차** — 여행 로그 페이지·상세를 **`/travel-log` 최상위 라우트**로 신설(계획 요약 자동 생성 + 사진 업로드/갤러리, 방당 최대 10장, 업로드 전 리사이즈/압축, **삭제 시 Storage 파일까지 제거**, 빈 상태 처리) — 지도 없이도 동작하는 핵심 가치. 내가 참여한 완료 여행만 조회되도록 스코프 적용
- **3차** — `SUPPORTED_COUNTRIES` 우선 국가 목록 확정(여행불가국 제외) → 세계지도(레벨 1, 홈 지도 수준 디테일) + 국가별 다트/뱃지, 다중 여행 시 방 이름 선택 팝업, 국가 다트 클릭 시 로그/리스트 진입
- **4차** — (기술 검증 먼저) `KoreaMapSVG` 단일→다중 마커 구조 검토 → 대한민국 상세지도 드릴다운(레벨 2) + 도 단위 다중 다트(+ 다중 여행 시 방 선택) + 크로스페이드 줌인 애니메이션
- **5차 (선택)** — 다른 국가 상세지도 순차 추가

> 지도 연출은 화려하지만 자산·구현 비용이 크다. **1~2차(완료 처리 + 로그 상세)만으로도 "여행 후 기록" 핵심 가치는 완성**되므로, 지도 드릴다운은 그 위에 얹는 순서를 권장.

## 부록: 검토자 의견 (확정본 시점)

- **여행 완료 버튼**은 홈 상단이 가장 자연스러움 (이미 이름/날짜 수정이 거기 있음)
- **세계지도 → 국가 줌인 → 상세지도** 컨셉은 몰입감이 뛰어나지만, 세계지도 자산·줌 애니메이션·국가별 상세지도가 모두 신규라 가장 무거운 파트임. 그래서 **로그의 핵심 가치(완료 처리 + 계획 요약 + 사진)를 먼저 완성**하고, 지도 드릴다운을 그 위에 단계적으로 얹는 순서로 확정.
- 지도 드릴다운은 **대한민국만 상세지도로 시작**, 다른 국가는 국가 다트에서 바로 로그로 진입 — 컨셉을 해치지 않으면서 초기 구현 부담을 크게 줄임.
- **국가·지역 저장은 "완료 시점"에** 받는 게 좋음 (계획 단계엔 목적지가 바뀔 수 있음)
- §5의 모든 항목이 결정 완료됨. 유일한 미확정 리스크는 **항목 4(국내 지역 다트 다중 표시)의 기술 실현 가능성** — 4차 착수 시 가장 먼저 `KoreaMapSVG` 구조를 검토해 결론을 낸다.
- (2차 검토에서 발견) **여행 로그 노출 범위**, **방장 이탈 시 권한 공백**, **동일 지역/국가 중복 여행 처리**는 구현 후 발견하면 되돌리기 번거로운 이슈라 기획 단계에서 미리 결정함(§5-9~11). 반면 **RLS 미적용에 따른 클라이언트 단 권한 체크 한계**는 앱 전체의 기존 트레이드오프와 동일한 성격으로 판단해 이번엔 별도로 해결하지 않기로 함.
- (3차 검토에서 발견/보정) 아래는 코드와 대조하며 새로 잡아낸 구현 함정들 — 기획서 본문에 반영 완료:
  1. **다트는 시/군이 아니라 도(道) 단위** — `KoreaMapSVG`가 도 좌표로만 다트를 찍으므로 컨셉의 "지역 다트"를 도 단위로 재정의(§2-2, §5-4·5·11)
  2. **`trip_members.is_admin` 기존 데이터 백필 필수** — 안 하면 기존 방들이 방장 없는 상태가 됨(§4)
  3. **`KOREA_REGIONS`에 대전·세종·대구·광주 없음** — 해당 도 여행을 로그에 못 남기는 공백, 보완 필요(§4, §5-12)
  4. **사진 삭제 시 Storage 파일까지 제거** — 기존 `AccommodationPage`는 DB 행만 지워 고아 파일이 쌓이는 선례, 용량이 동기인 만큼 처음부터 방지(§2-3)
  5. **`/travel-log`는 `TripLayout` 밖 최상위 라우트** → `useTripContext` 사용 불가, 자체 데이터 조회 필요(§2-1)
  6. **빈 상태·사진 삭제 권한·업로더 식별(`user.id`)** 명시(§2-2·2-3)
